◆ Hooks & headless automation

Feature 2.1.163 · Jun 4

Stop and SubagentStop hooks can return additionalContext

Until now a Stop or SubagentStop hook that wanted to push back got labeled a hook error. As of 2.1.163 those hooks can return hookSpecificOutput.additionalContext to hand Claude feedback and keep the turn going — a clean way to wire a verification step into the end of a run.

That's the mechanism behind a self-checking loop: a Stop hook runs your lint/build/test, and if it fails it feeds the error back and tells Claude to keep working instead of stopping with a red error.

So what for TSD: This is the one item with real leverage. Wire a Stop hook into your build agents that runs tsc / next build (or the report-manifest build on this very scout) and returns the failure as additionalContext — Claude fixes it in-loop instead of handing you a half-finished run to babysit.

Changelog 2.1.163
Fix 2.1.163 · Jun 4

claude -p no longer hangs after its result when a background command never exits

If a headless claude -p run left a backgrounded command alive (a dev server, a watcher), it would hang forever after printing its final result. Background shells are now stopped ~5s after the result once stdin closes.

So what for TSD: Directly relevant to your unattended runs — this daily scout and any cron-driven build agent. A run that spun up a Vite/Next dev server mid-task no longer wedges the whole job. Worth confirming your scheduled agents are on 2.1.163+.

Changelog 2.1.163
Fix 2.1.163 · Jun 4

Headless on Bedrock/Vertex/Foundry no longer demands ANTHROPIC_API_KEY under CI

claude -p was failing with "ANTHROPIC_API_KEY required" when CI=true and no Anthropic key was set, even on third-party providers that don't use one. Fixed.

So what for TSD: Only bites if you run headless Claude in a CI pipeline against Bedrock or Vertex. Not your stack today, but file it — it's the kind of thing that silently blocks a GitHub Action you'd add later.

◆ Plugins, skills & daily ergonomics

Improve 2.1.163 · Jun 4

/btw adds a "c to copy" shortcut

Press c after a /btw answer to copy the raw markdown to the clipboard, so headings, lists, and code blocks survive when you paste elsewhere.

So what for TSD: Small but daily — ask /btw a question mid-build, copy the clean markdown straight into a client email, a PR description, or a CHALLENGES.md entry without re-formatting.

Changelog 2.1.163
Feature 2.1.163 · Jun 4

/plugin list with --enabled/--disabled filters

A command to list installed plugins, optionally filtered to just the enabled or disabled ones.

So what for TSD: Inventory hygiene. When a skill or plugin misbehaves across your projects, /plugin list --enabled tells you what's actually live before you start debugging.

Changelog 2.1.163
Improve 2.1.163 · Jun 4

MCP & skill plumbing: session ID on resume, literal $ in commands

stdio MCP servers now get the same CLAUDE_CODE_SESSION_ID as hooks and Bash when you --resume, so a resumed session is no longer invisible to your MCP server. Skills also gained a \$ escape to put a literal $ before a digit in command bodies.

So what for TSD: If you build a custom MCP server that keys off the session ID (logging, per-session state), resume now works correctly. The \$ escape matters the next time a skill command needs a literal $1 in shell or a price string.
