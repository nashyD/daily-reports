◆ Where your tokens actually go

New 2.1.174

/usage now shows a full attribution breakdown

The VSCode Account & usage dialog (/usage) gained an attribution view: cache misses, long context, subagents, and a per-skill / per-agent / per-plugin / per-MCP breakdown over the last 24h or 7d.

You run heavy multi-agent workflows and 1M-context sessions all day. Until now "why did this burn so much" was guesswork. This makes it readable.

So what for TSD: Next time a Praxis or sales-dashboard session feels expensive, open /usage and look at the breakdown before you assume it's the model. If a single skill or MCP is the cost driver, you'll see it — that's the lever to trim. Worth a one-time baseline check across your active repos this week.
Changelog · 2.1.174
Fix 2.1.174

Workflow agent() subagents missing per-agent attribution headers

Subagents spawned inside the Workflow tool weren't carrying their own attribution headers. Now they do, so each agent() call lands in the breakdown above instead of being lumped together.

So what for TSD: When you fan out a workflow (review passes, migrations, your security-audit-style sweeps), you can now see which stage cost what. Pair it with signal #1 to decide whether a 16-agent fan-out is earning its tokens or you should scale it down.

◆ Model picker & background plumbing

Fix 2.1.174

/model picker now surfaces what Default actually resolves to

Three picker fixes: Opus now appears as its own row (Max/Team Premium/Enterprise), Sonnet on Pro/Team, Opus on pay-as-you-go API. A pinned ANTHROPIC_DEFAULT_SONNET_MODEL no longer shows a hardcoded Sonnet label. And the stale "Fable 5 is now consuming usage credits" banner stops showing for enterprise usage-based accounts.

So what for TSD: You're on Opus 4.8 1M — the picker now reflects that honestly instead of hiding it behind "Default." Minor, but it removes the "am I actually on the model I think I'm on" doubt when you switch repos.
Changelog · 2.1.174
Fix 2.1.174

Background sessions no longer inherit another session's provider env

Background sessions were picking up ANTHROPIC_* provider config (gateway URL, custom headers, /model aliases) from whatever shell started the background daemon. They now run with their own intended config.

So what for TSD: You lean on background agents and scheduled tasks (the daily-reports scout, the 7:23 sections run). This class of bug is exactly what would have a background run silently use the wrong model or endpoint. Update before you trust the next unattended run.
