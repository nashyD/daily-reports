◆ Keeping sessions moving

Feature 2.1.166

Configure up to three fallback models

A new fallbackModel setting lets you specify up to three models Claude Code tries in order when the primary is overloaded or unavailable. The existing --fallback-model flag now applies to interactive sessions, not just headless runs.

So what for TSD: Set a fallback chain in settings.json (e.g. Opus → Sonnet) so a Bisque or Studio C build session keeps moving when Opus is saturated at peak hours instead of erroring out and forcing a re-prompt. This is the single biggest throughput win in the release for a solo operator.
Changelog · 2.1.166
Improve 2.1.166

One retry on the fallback for unexpected errors

When the API rejects a turn with an unexpected non-retryable error, Claude Code now retries once on the fallback model. Auth, rate-limit, request-size, and transport errors still surface immediately.

So what for TSD: Fewer dead turns mid-build. During a long Next.js session you'll hit fewer hard stops that make you restart by hand — pairs directly with the fallback chain above.
Changelog · 2.1.166
Feature 2.1.166

Thinking can now be fully disabled

MAX_THINKING_TOKENS=0, --thinking disabled, and the per-model thinking toggle now disable thinking on models that think by default via the Claude API. Third-party providers are unchanged.

So what for TSD: For mechanical passes — bulk renames, find-and-replace across the Studio C repo — you can genuinely turn thinking off now to cut latency and token spend, where before the model kept thinking anyway.
Changelog · 2.1.166

◆ Permissions & agent security

Hardening 2.1.166

Relayed cross-session messages no longer carry user authority

Messages relayed via SendMessage from other Claude sessions no longer carry user authority. Receivers refuse relayed permission requests, and auto mode blocks them outright.

So what for TSD: If you run multiple sessions that message each other (or the daily-reports scout tasks), a relayed message can no longer approve its own permission prompts — closes a prompt-injection path. Nothing to change; just update and you're safer.
Changelog · 2.1.166
Feature 2.1.166

Glob patterns in deny rules

Deny rules now support glob patterns in the tool-name position — "*" denies all tools. Allow rules still reject non-MCP globs, and unknown tool names in deny rules now warn at startup.

So what for TSD: Lock a review or report-writing agent to read-only with a "*" deny plus a few targeted allows. Useful for the daily-reports scout and any client-facing agent you don't want writing to disk.
Changelog · 2.1.166
Fix 2.1.166

Managed-settings enforcement bugs

An invalid entry in managed settings no longer silently disables enforcement of the remaining valid policies. Separately, allowedMcpServers/deniedMcpServers predicates now match when they use ${VAR} references.

So what for TSD: Low priority solo, but worth knowing if you ever standardize a locked config across machines or a client's box — a single typo no longer quietly voids the rest of your policy.
Changelog · 2.1.166

◆ macOS & background-agent reliability

Fix 2.1.166

Orphaned bg-pty-host no longer pins a CPU core on macOS

Fixed orphaned claude --bg-pty-host processes spinning at 100% CPU after the daemon dies while connected on macOS.

So what for TSD: You're on macOS running scheduled background tasks (this scout included). This kills a silent 100%-CPU orphan that could drain battery and spin up the fans after a daemon crash — a real reason to take the update.
Changelog · 2.1.166
Fix 2.1.166

Two ways background sessions could get permanently wedged

Remote sessions no longer become permanently stuck when a brief backend disruption hits during worker registration at startup. And background agent sessions that entered a git worktree no longer crash-loop with "No conversation found" when reopened from claude agents.

So what for TSD: If you lean on background agents for parallel client work or worktree-isolated builds, these remove two failure modes that could leave a session dead and need a manual kill.
Changelog · 2.1.166
Fix 2.1.166

The recurring "image could not be processed" error

Fixed a recurring "image could not be processed" error and the extra token usage that came with it when an unprocessable image was sent in a session.

So what for TSD: If you paste screenshots into sessions (design reviews, client mockups), you'll stop burning tokens on a silent retry loop when one image doesn't decode.
