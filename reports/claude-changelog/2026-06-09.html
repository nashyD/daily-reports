2.1.169 is a real release, not another bug-fix patch

After 2.1.165–2.1.168 shipped almost nothing but reliability fixes, 2.1.169 brings three things worth changing your habits for: a /cd command that moves a session between repos without nuking the prompt cache, a --safe-mode boot that disables every customization at once, and a setting to hide bundled skills from the model. The rest is a long tail of agent, security, and macOS fixes.

◆ Multi-repo workflow

New 2.1.169

/cd moves a session to a new directory without breaking the cache

The new /cd command repoints the current session at another working directory and keeps the prompt cache warm, so you don't eat a full cold context re-read just to cross a repo boundary.

So what for TSD: Your days hop between repos — tsd-modernization-app, praxis, daily-reports, gallantrenters. Until now that meant a fresh claude per repo or a cache-busting restart. Use /cd to carry context across, e.g. from the sales-dashboard work into the portal-passkeys repo, without paying the cache miss each time.

◆ Debugging your own setup

New 2.1.169

--safe-mode and disableBundledSkills give you two new control surfaces

--safe-mode (or CLAUDE_CODE_SAFE_MODE) starts Claude Code with all customizations disabled at once — CLAUDE.md, plugins, skills, hooks, MCP servers — for troubleshooting. Separately, the new disableBundledSkills setting (and CLAUDE_CODE_DISABLE_BUNDLED_SKILLS) hides bundled skills, workflows, and built-in slash commands from the model.

So what for TSD: Your environment is heavily customized — dozens of advisor skills, scheduled tasks, MCP servers, a big memory layer. When something misbehaves, --safe-mode is the one-flag bisect: does it still break with everything off? And disableBundledSkills trims the huge built-in skill/command menu out of the model's context so it isn't distracted by skills you never use on a given task.

◆ Agents & background reliability

Fix 2.1.169

Background agents stop ignoring your project env and flags

Background agents dispatched onto a pre-warmed worker were ignoring project-level settings env values like ANTHROPIC_MODEL; that's fixed. They also now preserve --ide, --chrome, --bare, --remote-control and other flags across retire→wake, and are told upfront that shared-checkout edits are blocked until they enter a worktree. claude agents --json no longer omits blocked and just-dispatched sessions, gained --all for completed sessions, and added id and state fields.

So what for TSD: Your daily scout reports and any background dispatch you lean on will now actually run on the model you pinned in settings, instead of silently booting on the daemon's default. If you script around claude agents --json, the new id/state fields make status checks reliable.
Claude Code Changelog · 2.1.169
Fix 2.1.169

Stale permission prompts no longer haunt resumed remote sessions

Reattaching to a remote session whose worker had died while waiting on a prompt used to resurface the same stale permission and dialog prompts every time. Remote Control also no longer gets stuck on "reconnecting" when an OAuth token refresh lands during a resume.

So what for TSD: If you drive longer-running sessions from your phone or a second machine, reattaching is less likely to dump you into a pile of dead prompts or a hung reconnect.
Claude Code Changelog · 2.1.169

◆ Security & enterprise settings

Fix 2.1.169

Untrusted project settings can no longer set OTEL client-cert paths silently

A project's untrusted settings could previously set OpenTelemetry client-certificate paths without a trust confirmation. That's now gated. Enterprise managed MCP policies (allowedMcpServers/deniedMcpServers) that weren't being enforced on reconnect, on IDE-typed configs, or before remote settings loaded are now enforced; and remote-managed settings with one invalid entry apply their remaining valid policies instead of silently dropping the whole payload.

So what for TSD: This is the same class of issue your June fleet security review chased — config from a cloned repo quietly changing where credentials or telemetry point. Worth noting as a reason to keep Claude Code current across the machines you run client work on.
