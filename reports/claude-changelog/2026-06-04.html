◆ Headline capabilities

New 2.1.154

Dynamic workflows orchestrate hundreds of agents

You can now ask Claude to build a workflow that spreads a task across tens to hundreds of agents running in the background, then watch them with /workflows. This is the structural shift in the release: complex jobs stop being one long single-agent session and become a fan-out you supervise.

So what for TSD: a one-person delivery shop can now run audits, migrations, and multi-page builds in parallel instead of serially. This is the single feature most likely to change your throughput on a client build.
changelog ◆ 2.1.154
Model 2.1.154 / 2.1.158

Opus 4.8 lands as the default, with an effort ceiling

Opus 4.8 now defaults to high effort, with /effort xhigh reserved for the hardest problems and a leaner system prompt cutting noise. Fast mode runs at 2× cost for ~2.5× speed when you need a quick pass.

Four releases later, Auto mode for Opus 4.7 and 4.8 reached Bedrock, Vertex, and Foundry — opt in with CLAUDE_CODE_ENABLE_AUTO_MODE=1.

So what for TSD: default quality went up without you touching a setting. Keep Fast mode for throwaway edits; spend xhigh on the gnarly client logic where a bug costs a callback.
changelog ◆ 2.1.154changelog ◆ 2.1.158

◆ Workflow & delivery upgrades

Improved 2.1.152

Code review now applies its own fixes

/code-review --fix writes its reuse, simplification, and efficiency findings straight into the working tree, and /simplify now runs that cleanup pass for you. Review stopped being a list of suggestions you action by hand.

So what for TSD: the part of solo delivery with no second engineer — the review pass — is now one command. Run it before every client push.
changelog ◆ 2.1.152
New 2.1.157

Project-local skills and plugins load on their own

Plugins placed in .claude/skills auto-load with no marketplace step, claude plugin init <name> scaffolds a new one, and defaultEnabled: false lets a plugin ship opt-in. EnterWorktree can also switch between Claude-managed worktrees mid-session now.

So what for TSD: bespoke per-client skills (a Bisque tone guide, a Studio C booking helper) can live in each repo and just work. Standardize your build skills per project.
changelog ◆ 2.1.157
New 2.1.154 / 2.1.162

Background shell sessions and a clearer agent dashboard

Inside claude agents, typing ! <command> spawns a background shell you can attach to and detach from — long builds and test runs stop blocking the session. The dashboard now shows done/total for fanned-out work and surfaces a waitingFor field via claude agents --json so you can see what's blocking a session.

So what for TSD: kick off a Vercel build or a slow test suite and keep working. With multiple client repos in flight, the dashboard is how you stop losing track of sessions.
changelog ◆ 2.1.154changelog ◆ 2.1.162
Safety 2.1.160

Guardrails before touching shell + build config

Claude now prompts before writing to shell startup files (.zshenv, .zlogin, .bash_login), and acceptEdits mode asks before editing build-tool config. Edits also no longer require a separate Read after a single-file grep.

So what for TSD: safer to run in higher-autonomy modes on a live client repo without it quietly rewriting your environment.
changelog ◆ 2.1.160
