◆ Agents and automation

Feature 2.1.172

Sub-agents can spawn their own sub-agents, up to 5 levels deep

Until now an agent could fan out to a flat layer of workers. As of 2.1.172 a sub-agent can itself dispatch sub-agents, and so on, five levels down. That turns orchestration from "one boss, many workers" into a real tree — a coordinator can split a job, hand each branch to a lead, and let that lead break its branch down further.

So what for TSD: Your bigger jobs — a full audit across the Bisque / Studio C / Sonderwerks repos, or a multi-file migration — can now be expressed as a tree instead of one giant flat fan-out you babysit. Use it deliberately: nesting multiplies token spend fast, so cap depth to the job and lean on it for genuinely hierarchical work, not every task.

Fix 2.1.172

Background agents no longer read the wrong directory's project settings

When a background agent was dispatched onto a pre-warmed worker, it could read a different directory's project settings — .mcp.json approvals, trust state — instead of the project it was actually working in. 2.1.172 fixes that, along with a background sub-agent staying stuck as "active" after a nested agent it spawned was stopped, and background-session attach failing with EAUTH for sessions started on an older version after the daemon auto-updated.

So what for TSD: This is the one to care about if you run background agents across multiple client repos in a sitting. The old behavior meant a worker warmed in one repo could silently carry another repo's MCP approvals — a quiet trust leak. Update before your next multi-repo background run.

◆ Your daily driver — models and context

Fix 2.1.172

1M-context sessions without credits auto-compact instead of locking up

A session running 1M context that ran out of usage credits used to get permanently stuck — no recovery short of abandoning it. Now the session automatically compacts back under the standard context limit and keeps going. The same release also fixed model IDs picking up a doubled suffix like [1M][1m] when ANTHROPIC_DEFAULT_OPUS_MODEL already carried one, and restored Opus/Sonnet 1M rows in the /model picker for orgs using version-specific IDs.

So what for TSD: You run Opus 4.8 on 1M context as your default. This is the failure mode that could have eaten a long working session mid-task. After updating, a credit ceiling degrades gracefully instead of bricking the session.

Fix 2.1.173

Fable 5 [1m] suffix is now stripped automatically

Fable 5 includes 1M context by default, so model names carrying a [1m] suffix were redundant. 2.1.173 normalizes them — the suffix is stripped rather than treated as a distinct, possibly unrecognized model string. The same patch quiets a spurious "sandbox dependencies missing" startup warning on Windows.

So what for TSD: Minor housekeeping that happens to touch your exact setup — your model ID surfaces as claude-opus-4-8[1m], the same suffix pattern. If you ever pin Fable 5 in a script or settings file, you can drop the suffix and trust it resolves cleanly.

◆ Performance, footprint, and review

Improve 2.1.172

Long conversations stream cheaper; idle CPU drops

2.1.172 removes redundant message normalization and skips full message-history transforms when streaming tool-use state hasn't changed — a direct win in long sessions. Idle cost is down too: the /goal status chip no longer repaints the terminal at 5 Hz while idle, parallel subagents trigger fewer re-renders, and Claude in Chrome now loads its browser tools in one batched call instead of one per tool.

So what for TSD: You live in long, agent-heavy sessions. These are the kind of changes you feel as a quieter fan and a more responsive terminal during big builds rather than a headline feature. No action — just update and benefit.

Claude Code changelog — 2.1.172
Improve 2.1.172

/code-review ultra stays available when logged out; WebFetch wildcard rules fixed

/code-review now keeps the ultra option visible even when you're not signed in to claude.ai, with an explanation that the cloud review needs an account. Separately, WebFetch(domain:*.example.com) wildcard rules never matched subdomains in allow/deny/ask position, and file rules with mid-pattern wildcards like Read(secrets-*/config.json) were rejected at startup — both fixed.

So what for TSD: You use /code-review ultra on branches before merge — it no longer vanishes from the menu if your claude.ai session lapsed; it tells you to sign in. And if you ever wrote a wildcard WebFetch or file-permission rule that "wasn't working," this is why — they match now.
